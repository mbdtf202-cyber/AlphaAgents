# AlphaAgents MVP Strategy Operating Model

## 1. MVP objective

AlphaAgents is building the full Agent-native transaction and managed-delivery platform, but the MVP must prove one narrow commercial loop first: a buyer can purchase a bounded Agent-delivered result, review evidence, complete acceptance, and move through finance and reputation gates without ambiguity.

The MVP is not a generic marketplace launch. It is a trial-first, procurement-grade operating lane that proves the platform can turn one real demand pattern into repeatable revenue and replayable evidence.

## 2. Default MVP motion

- Default first purchase: `跨境电商竞品监控与内容选题情报包`.
- Default first ICP: `US TikTok Shop beauty and personal-care`.
- Default motion: `Trial Quick Order -> acceptance or bounded revision -> Standard / Pro repeat order -> PoC -> annual order-credit or managed program`.
- Default operating boundary: read-only public-source research or customer-uploaded copies only.

The first MVP order must stay narrow enough to verify delivery quality quickly:

- frozen scope before payment
- explicit acceptance owner
- evidence index and QA before buyer review
- conditional release rather than unconditional payout
- reputation writeback only after acceptance or adjudicated dispute

## 3. Trial-first go-to-market

Trial is the default first commercial action because it shortens buyer learning time and forces the platform to prove structure, not just narrative.

Trial-first means:

- no broad multi-seller shopping flow before the first bounded purchase
- no promise of autonomous account login, publishing, ad spend, or funds movement in the initial lane
- no enterprise-procurement claim until real contracting, invoicing, refund path, and subprocessor details are complete
- no upgrade to PoC or annual path unless Trial quality, evidence completeness, and buyer workflow discipline are already proven

This creates a clean commercial ladder:

1. Trial validates speed, evidence readability, and acceptance behavior.
2. Standard and Pro validate repeatability, margin, and reduced ops overhead.
3. PoC validates multi-order orchestration, finance controls, and procurement cooperation.
4. Program / order-credit validates long-term retention and category economics.

## 4. AaaS boundaries

AlphaAgents is Agent as a Service, not seat-first SaaS and not an app store that lets supply bypass accountability.

Boundary rules for the MVP:

- Agent Apps, custom Agents, and managed services still use the same identity, permission, evidence, acceptance, finance, and reputation chain.
- Subscription, usage, or installation can define access and billing, but cannot replace execution evidence or acceptance proof.
- The platform sells accountable results and controlled execution, not unsupervised tool access.
- Buyer-facing language must describe the transaction as a managed result with evidence and acceptance, not as an AI feature bundle.

## 5. Command, UI, API, runtime consistency

The MVP only counts if all four surfaces describe and mutate the same transaction truth:

- UI explains the buyer path and exposes the current frozen state.
- CLI provides the same core actions and state transitions.
- API carries the same contract and validation rules.
- Agent runtime emits execution and delivery artifacts that map back to the same records.

Required consistency rules:

- every core write uses the shared command contract
- no UI-only shortcut for payment, acceptance, dispute, or listing changes
- no runtime callback that bypasses the same permission and event rules
- no buyer-facing state that cannot be replayed from machine-readable records

## 6. Evidence and finance gates

The MVP operating model is evidence-first and finance-aware.

Evidence gates:

- every first-order delivery must produce a structured evidence package
- sandbox material must stay marked `sample_only + sandbox_verified`
- validated business claims require real payment, acceptance, and delivery proof
- QA failure blocks buyer acceptance

Finance gates:

- payment confirmation is required before execution enters funded delivery
- buyer-facing terminology is `conditional release` workflow
- internal escrow-style ledger states must not be marketed as licensed clearing capability
- payout, refund, and partial release decisions must remain attached to frozen terms and evidence-backed acceptance outcomes

## 7. Rollout phases

### Phase 0: Sandbox verification

Goal: prove the artifact structure, event replay, and acceptance/finance paths in sample form.

Exit signal:

- sandbox packages open cleanly
- verifier scripts pass
- UI, CLI, API, and runtime snapshots agree on the same sample order

### Phase 1: Trial launch

Goal: win and deliver the first real Trial orders in the default ICP.

Exit signal:

- real payment captured
- buyer acceptance or bounded revision completed
- evidence and finance artifacts retained

### Phase 2: Repeat-order proof

Goal: convert Trial into Standard, Pro, or repeated category work with lower operational friction.

Exit signal:

- at least one repeat buyer path exists
- acceptance, payout, refund, and reputation flows remain consistent
- unit economics improve relative to Trial-only motion

### Phase 3: Program expansion

Goal: expand from single-order execution into PoC, order-credit, or managed program operations without breaking the same control model.

Exit signal:

- procurement pack is complete for the target buyer
- finance and risk roles can operate from the same evidence trail
- category expansion happens only after the first lane is operationally stable

## 8. Acceptance posture

This MVP document does not replace the broader platform design. It defines the operating lane that must become real first.

Acceptance for MVP readiness should ask:

- Is the first transaction narrow, honest, and sellable?
- Can the buyer understand what they are buying, how it will be reviewed, and when funds move?
- Can the platform replay the same truth across command, UI, API, runtime, evidence, and finance?
- Does expansion happen only after the default lane proves repeatable?
